Role mapping: Cream for page backgrounds and muted surfaces. Sky and Sky-soft for accents and data fills. Slate for secondary UI, borders-on-tint, and supporting labels. Deep for primary actions, active nav, and key data. Ink for top bars, footers, and code surfaces. Semantic colors are reserved for status communication — never decorative.

2. Typography

Inter / system-ui for UI, JetBrains Mono for code. Two weights only (400 / 500). Sentence case everywhere — never Title Case, never ALL CAPS (except ≤11px meta labels).

Display — 28px / 500

Heading — 20px / 500

Subheading — 15px / 500

Body — 13px / 400, line-height 1.6. Optimised for dense dashboards and editorial CMS contexts where legibility at small sizes matters more than visual flourish.

3. Project icons

One icon per project across La Sentinelle Group. Each icon sits in a 96px rounded-square ink frame with palette-derived glyphs, and exports cleanly to the 16/32/64/128/512px favicon set. SVG source is theme-agnostic — the frame color stays fixed so favicons and app-switcher tiles remain recognisable on both light and dark browser chrome.

Construction rules. All icons share a 96×96 viewBox, a 20px corner radius, and an ink background (#0F2C44) so they read as a coherent set across the group. Cream (#F1F6F8) is the primary glyph color; Sky, Slate, and Deep build the secondary layer and signify brand, state, or depth. Each icon carries a visual metaphor aligned with its product — a reservation grid for Digital Bookings, a tablet-reading moment for lexpress.mu, an armored sentinel standing watch for lasentinelle.mu (literal "sentinel"), and stacked volumes with a clock for the archive. For favicons, export PNG at 16, 32, 48, 64, 128, 192, 256, 512 and generate an ICO bundle containing the first three sizes.

18. Destructive action confirmations

Every destructive action across the backoffice goes through this modal — never a native confirm() prompt, never a bare button. Two tiers by reversibility: amber warn for reversible operations (cancel, soft-delete, status change), red danger with typed-match gate for permanent deletes and cascades.

Archive this author?

Existing articles keep the byline, but the author can no longer be assigned to new pieces.

Marie Claire
Senior reporter, politics desk · 142 articles

Remove this user?

They will lose access immediately. Any articles or photos they own remain in place but they will no longer be able to sign in. This action cannot be undone.

Jean Dupont (jean@lexpress.mu)
Type jean@lexpress.mu to confirm.

Implementation

Use the <x-confirm-delete> Blade component anywhere a row or detail view exposes a delete action. Each instance gets its own scoped Alpine modal state and teleports to <body>, so it overlays correctly even from inside a table cell. Production overlays wrap the dialog in .ls-overlay (fixed-position scrim that fills the viewport); the .dialog-mock wrapper above is a documentation-only container that lets the doc page show two dialogs side-by-side.
<x-confirm-delete :action="route('articles.destroy', $article)" :name="$article->title" title="Delete article?" description="Comments and updates attached to this article will be removed too." :requireTyping="$article->slug" />

When to require typing

Reach for :requireTyping only when the action's blast radius warrants the extra friction. Default to a plain confirm for routine deletes — over-using the gate trains users to ignore it.
Action Variant Type-to-confirm phrase
Delete category, tag, photographer danger none — name in dialog-target is enough
Soft-delete an author or briefing warn none
Remove a user account danger email address
Delete a published article danger article slug
Bulk delete (any cascade > 10 rows) danger "DELETE" literal

Copy guidelines

  • Title is a question ("Delete category?") — not a statement. Forces the user to answer.
  • Description states the consequence in plain terms. Always include "cannot be undone" (in <strong>) for hard deletes.
  • Confirm label is a verb that matches the title's intent ("Delete permanently", "Archive permanently", "Remove user") — never a generic "OK" or "Yes".
  • Cancel label stays "Cancel". Don't get cute.
  • Don't apologise. The user asked for this; tell them what will happen and let them decide.
